# RuoYi-Vue 服装ERP项目编码规范

## 1. 总则

### 1.1 目的

为了提高代码质量，规范代码风格，提升团队协作效率，特制定本编码规范。

### 1.2 适用范围

本规范适用于 RuoYi-Vue 服装ERP项目的所有开发人员。

### 1.3 参考标准

- 阿里巴巴Java开发手册
- 前端开发规范
- 数据库设计规范

## 2. Java后端编码规范

### 2.1 命名规范

#### 2.1.1 包名

- 包名使用小写字母，单词间用点分隔
- 域名倒写 + 项目名 + 模块名
- 示例：
  ```java
  com.ruoyi.demo
  com.ruoyi.demo.controller
  com.ruoyi.demo.service
  com.ruoyi.demo.mapper
  com.ruoyi.demo.domain
  ```

#### 2.1.2 类名

- 使用大驼峰命名法
- 类名要具有描述性，能体现类的用途
- 示例：
  ```java
  DemoOrder  // 订单类
  DemoOrderController  // 订单控制器
  DemoOrderService  // 订单服务
  DemoOrderMapper  // 订单数据访问接口
  ```

#### 2.1.3 方法名

- 使用小驼峰命名法
- 方法名要动词开头，具有描述性
- 示例：
  ```java
  getOrderById()  // 根据ID获取订单
  saveOrder()  // 保存订单
  deleteOrder()  // 删除订单
  ```

#### 2.1.4 变量名

- 使用小驼峰命名法
- 变量名要具有描述性，避免使用缩写
- 示例：
  ```java
  orderNo  // 订单号
  customerName  // 客户名称
  productPrice  // 产品价格
  ```

#### 2.1.5 常量名

- 使用全大写字母，单词间用下划线分隔
- 示例：
  ```java
  MAX_ORDER_COUNT = 1000;  // 最大订单数量
  DEFAULT_PAGE_SIZE = 20;  // 默认每页大小
  ```

### 2.2 代码格式

#### 2.2.1 缩进

- 使用4个空格缩进
- 禁止使用Tab字符

#### 2.2.2 换行

- 每行代码长度不超过120个字符
- 过长的代码需要适当换行
- 运算符放在行首

#### 2.2.3 空行

- 在类、方法、代码块之间适当添加空行
- 逻辑相关的代码块之间可以添加空行

### 2.3 注释规范

#### 2.3.1 文件头部注释

- 每个Java文件头部需要添加版权信息和作者信息
- 示例：
  ```java
  /**
   * 订单控制器
   *
   * @author ruoyi
   * @date 2026-04-01
   */
  @RestController
  @RequestMapping("/demo/order")
  public class DemoOrderController extends BaseController {
      // ...
  }
  ```

#### 2.3.2 类注释

- 每个类需要添加类注释，描述类的功能
- 示例：
  ```java
  /**
   * 订单表
   *
   * @author ruoyi
   * @date 2026-04-01
   */
  @Data
  @TableName("demo_order")
  public class DemoOrder extends BaseEntity {
      // ...
  }
  ```

#### 2.3.3 方法注释

- 每个方法需要添加注释，描述方法的功能和参数
- 示例：
  ```java
  /**
   * 获取订单列表
   *
   * @param order 订单条件对象
   * @return 订单列表
   */
  @GetMapping("/list")
  public TableDataInfo list(DemoOrder order) {
      // ...
  }
  ```

#### 2.3.4 字段注释

- 重要字段需要添加注释，描述字段的含义
- 示例：
  ```java
  /** FOB价格 */
  private BigDecimal fobPrice;

  /** CIF价格 */
  private BigDecimal cifPrice;
  ```

### 2.4 代码质量要求

#### 2.4.1 避免魔法数字

- 禁止在代码中直接使用魔法数字
- 应该使用常量或枚举替代
- 示例：
  ```java
  // 错误写法
  if (status == 1) {
      // ...
  }

  // 正确写法
  if (status == OrderStatus.PENDING.getValue()) {
      // ...
  }

  public enum OrderStatus {
      PENDING(1, "待处理"),
      PROCESSING(2, "处理中"),
      COMPLETED(3, "已完成");
  }
  ```

#### 2.4.2 避免重复代码

- 相同或相似功能的代码需要进行抽象和封装
- 使用工具类或公共方法
- 示例：
  ```java
  // 公共工具方法
  public static BigDecimal calculateProfit(BigDecimal revenue, BigDecimal cost) {
      return revenue.subtract(cost);
  }

  // 使用公共方法
  BigDecimal profit = OrderUtils.calculateProfit(order.getRevenue(), order.getCost());
  ```

#### 2.4.3 异常处理

- 异常处理要适当，不要捕获异常后什么都不做
- 异常信息要具体，包含错误原因
- 示例：
  ```java
  // 错误写法
  try {
      // ...
  } catch (Exception e) {
      // 什么都不做
  }

  // 正确写法
  try {
      // ...
  } catch (Exception e) {
      log.error("获取订单失败: {}", e.getMessage(), e);
      throw new ServiceException("获取订单失败，请稍后重试");
  }
  ```

## 3. Vue前端编码规范

### 3.1 文件和目录命名

- 文件和目录使用kebab-case命名法
- 示例：
  ```
  src/views/demo/
  ├── order/
  │   ├── index.vue
  │   └── order-detail.vue
  ├── outsource/
  │   ├── index.vue
  │   └── outsource-list.vue
  ```

### 3.2 组件命名

- 组件使用PascalCase命名法
- 组件名要具有描述性
- 示例：
  ```vue
  <!-- OrderList.vue -->
  <template>
    <div>
      <!-- 订单列表组件 -->
    </div>
  </template>

  <script>
  export default {
    name: 'OrderList'
  }
  </script>
  ```

### 3.3 代码格式

- 使用4个空格缩进
- 每行代码长度不超过120个字符
- 适当的空行和换行

### 3.4 注释规范

- 使用HTML注释或JavaScript注释
- 重要的逻辑需要添加注释
- 示例：
  ```vue
  <template>
    <!-- 订单列表 -->
    <div class="order-list">
      <!-- 搜索栏 -->
      <div class="search-bar">
        <!-- ... -->
      </div>
      <!-- 数据表格 -->
      <el-table :data="orderList">
        <!-- ... -->
      </el-table>
    </div>
  </template>

  <script>
  /**
   * 订单列表组件
   *
   * 展示订单列表，支持搜索和分页
   */
  export default {
    name: 'OrderList',
    data() {
      return {
        orderList: [],
        searchForm: {}
      }
    },
    methods: {
      /**
       * 获取订单列表
       */
      async getOrderList() {
        try {
          const response = await this.$http.get('/demo/order/list', {
            params: this.searchForm
          });
          this.orderList = response.data.rows;
        } catch (error) {
          this.$message.error('获取订单列表失败');
        }
      }
    }
  }
  </script>
  ```

## 4. 数据库设计规范

### 4.1 命名规范

- 表名使用小写字母，单词间用下划线分隔
- 字段名使用小写字母，单词间用下划线分隔
- 示例：
  ```sql
  CREATE TABLE demo_order (
      id BIGINT PRIMARY KEY,
      order_no VARCHAR(50),
      customer_name VARCHAR(100),
      product_price DECIMAL(10,2),
      order_status VARCHAR(20)
  );
  ```

### 4.2 字段类型

- 主键使用BIGINT类型
- 字符串字段使用VARCHAR类型
- 金额字段使用DECIMAL类型，保留2位小数
- 日期字段使用DATE或DATETIME类型

### 4.3 索引设计

- 主键自动创建索引
- 常用查询字段需要创建索引
- 外键字段需要创建索引

## 5. Git规范

### 5.1 分支管理

- 使用Git Flow分支管理模型
- 主分支：main（生产环境代码）
- 开发分支：develop（集成测试代码）
- 功能分支：feature/*（新功能开发）
- 修复分支：hotfix/*（紧急修复）

### 5.2 提交规范

- 提交信息格式：`[模块名] 功能描述 [任务号]`
- 示例：
  ```
  [订单管理] 实现订单列表查询功能 [TASK-001]
  [外发加工] 添加外发单状态字段 [TASK-002]
  [生产排程] 修复排程算法错误 [BUG-001]
  ```

### 5.3 代码审查

- 每个功能开发完成后需要进行代码审查
- 至少需要2名团队成员进行审查
- 审查通过后才能合并到主分支

## 6. 测试规范

### 6.1 单元测试

- 每个方法需要编写单元测试
- 覆盖率要求：核心业务方法覆盖率不低于80%
- 使用JUnit和Mockito进行测试

### 6.2 集成测试

- 接口测试使用Postman或Swagger进行
- 数据库操作测试使用H2数据库

### 6.3 系统测试

- 功能测试使用Selenium进行自动化测试
- 性能测试使用JMeter进行压力测试

## 7. 文档规范

- 使用Markdown格式编写

## 8. 代码审查规范

### 8.1 审查要点

- 代码是否符合命名规范
- 代码是否符合格式规范
- 代码是否符合质量要求
- 逻辑是否正确
- 是否有潜在的性能问题
- 是否有安全漏洞

### 8.2 审查流程

- 开发人员完成功能后，提交Pull Request
- 至少2名团队成员进行审查
- 审查人员提出修改建议
- 开发人员修改后，再次进行审查
- 审查通过后，合并到主分支

### 8.3 审查工具

- 使用Git平台的代码审查功能
- 使用SonarQube进行代码质量检测

通过遵循以上编码规范，可以提高代码质量，减少代码缺陷，提升团队协作效率，确保项目的可维护性和可扩展性。
