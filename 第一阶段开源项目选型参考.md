# 第一阶段：开源项目选型参考清单

> 本文档为「叫车业务微信群运营数字化」项目第一阶段（微信群信息自动化处理系统）的开源项目调研结果，供技术选型参考。

---

## 一、项目分类总览

| 类别 | 用途 | 推荐度 | 适用场景 |
|------|------|--------|---------|
| **A: 企业微信Java SDK** | 企业微信API全量接入（消息收发/通讯录/机器人等） | ⭐⭐⭐⭐⭐ | 与现有Spring Boot ERP无缝集成 |
| **B: 会话存档SDK** | 读取企业微信群聊消息内容（核心需求） | ⭐⭐⭐⭐⭐ | 消息监听与数据提取 |
| **C: Webhook机器人SDK** | 向企业微信群自动发送消息（图表/清单推送） | ⭐⭐⭐⭐⭐ | 自动发送功能 |
| **D: AI机器人框架** | 消息智能处理 + 流式回复 + 规则引擎 | ⭐⭐⭐⭐ | 数据提取规则引擎 |
| **E: Python生态方案** | 快速原型开发 / 独立微服务 | ⭐⭐⭐ | 轻量级快速实现 |

### A类：企业微信 Java SDK（推荐 - 与ERP同栈）

#### 1. wecom-sdk ⭐⭐⭐⭐⭐ 强烈推荐

| 项 | 内容 |
|---|------|
| **仓库地址** | [gitee.com/felord/wecom-sdk](https://gitee.com/felord/wecom-sdk) |
| **语言** | Java |
| **Stars** | 高（Gitee热门项目） |
| **维护状态** | 活跃（近3年持续迭代） |
| **许可证** | Apache 2.0 |

**功能覆盖（200+ API）：**

| 模块 | 支持的接口 | 与本项目的关联 |
|------|-----------|--------------|
| **消息推送** | 应用消息、群机器人消息发送 | ✅ 自动发送清单到群 |
| **企微机器人** | 机器人创建、管理、Webhook | ✅ 机器人收发消息 |
| **通讯录管理** | 部门、成员、标签管理 | ✅ 司机档案同步 |
| **身份验证** | access_token管理 | ✅ 基础能力 |
| **应用管理** | 应用配置、权限 | ✅ 集成基础 |

**集成方式（Spring Boot）：**

```xml
<!-- pom.xml -->
<dependency>
    <groupId>com.github.felord</groupId>
    <artifactId>wecom-sdk</artifactId>
    <version>最新版本</version>
</dependency>
```

```java
// 示例：发送群机器人消息
@Autowired
private WeComRobotClient robotClient;

public void sendToGroup(String webhookUrl, String content) {
    TextMessage message = TextMessage.of(content);
    robotClient.send(webhookUrl, message);
}
```

**为什么强烈推荐：**
- ✅ **纯Java实现**，与现有RuoYi ERP（Spring Boot）完美兼容
- ✅ **200+接口全覆盖**，一套SDK解决所有企微集成需求
- ✅ **文档完善**，社区活跃，问题容易找到答案
- ✅ **模块化设计**，按需引入依赖，不臃肿
- ✅ **持续维护**，3年迭代，质量有保障

#### 2. wework-webhook-starter ⭐⭐⭐⭐⭐ 强烈推荐

| 项 | 内容 |
|---|------|
| **仓库地址** | GitHub/Gitee搜索 `wework-webhook-starter` |
| **语言** | Java (Spring Boot Starter) |
| **定位** | 企业微信群机器人Webhook SDK |

**核心能力：**

- 三行代码接入，开箱即用
- 支持多机器人实例配置
- 支持**文本、Markdown、图文卡片(News)**三类消息
- Spring Boot自动装配，零配置启动

**集成示例：**

```xml
<dependency>
    <groupId>io.github.dengliming</groupId>
    <artifactId>wework-webhook-starter</artifactId>
    <version>最新版本</version>
</dependency>
```

```yaml
# application.yml
wework:
  webhook:
    robots:
      - name: 叫车通知群
        url: https://qyapi.weixin.qq.com/cgi-bin/webhook/send?key=xxx
      - name: 日报群
        url: https://qyapi.weixin.qq.com/cgi-bin/webhook/send?key=yyy
```

```java
@Autowired
private MessageService messageService;

// 发送Markdown格式的叫车清单
messageService.send("叫车通知群", MarkdownMessage.builder()
    .content("## 今日叫车清单\n| 时间 | 路线 | 货物 |\n|---|---|---|\n| 14:00 | 富泉→宏发 | 面料500kg |")
    .build());
```

**适用场景：** 第一阶段的「自动发送模块」——将生成的叫车清单/图表推送到指定群

### B类：会话存档SDK（消息读取 - 核心依赖）

#### 3. WeWorkFinanceSDK (Go版) ⭐⭐⭐⭐⭐

| 项 | 内容 |
|---|------|
| **仓库地址** | [github.com/NICEXAI/WeWorkFinanceSDK](https://github.com/NICEXAI/WeWorkFinanceSDK) |
| **语言** | Go |
| **定位** | 企业微信会话存档官方C版SDK的Go封装 |
| **许可证** | 开源 |

**核心能力：**

- 基于**企业微信官方C版SDK**封装（合规）
- 提供**消息解密、数据拉取、媒体文件下载**等完整能力
- Linux环境友好，适合部署在服务器上
- 清晰的Go API接口

**使用前提：**
- 企业微信已开通「会话存档」功能
- 已获取相关密钥（corp_id/secret/secret_key等）
- 成员已签署知情同意书

**架构位置（在本项目中）：**

```
┌──────────────────────┐
│   WeWorkFinanceSDK   │  ← 独立Go微服务
│   (消息解密+拉取)    │
└──────────┬───────────┘
           │ 解密后的结构化数据
           ▼
┌──────────────────────┐
│  Spring Boot 后端     │  ← 现有ERP扩展
│  (规则引擎+数据处理)  │
└──────────────────────┘
```

> **建议：** 将会话存档作为独立Go微服务部署，通过HTTP/gRPC与主系统通信。这样既满足官方SDK的语言要求，又不影响现有Java技术栈。

### D类：AI机器人 / 智能处理框架

#### 6. wecom_aibot_sdk (Python) ⭐⭐⭐⭐

| 项 | 内容 |
|---|------|
| **仓库地址** | [github.com/chengyongru/wecom_aibot_sdk](https://github.com/chengyongru/wecom_aibot_sdk) |
| **语言** | Python |
| **定位** | 企业微信智能机器人SDK（WebSocket长连接） |

**核心亮点：**

- 基于**WebSocket长连接**，无需公网IP和域名！
- 自动重连（指数退避策略）
- **消息分发机制**：自动解析消息类型并触发事件回调
- **流式回复**：支持Markdown和混合内容
- 支持模板卡片、文件下载解密

**事件驱动模型：**

```python
from wecom_aibot_sdk import WeComAibot

bot = WeComAibot(client_id="...", client_secret="...")

@bot.on_message("text")
def handle_text(message):
    # 处理文本消息 → 提取叫车信息
    data = extract_car_order(message.content)
    return f"收到叫车订单：{data}"

@bot.on_message("image")
def handle_image(message):
    # 处理图片消息 → OCR识别
    text = ocr_recognize(message.image_url)
    return process_ocr_result(text)

bot.run()
```

**为什么值得关注：**
- ✅ **WebSocket模式无需公网IP**，大大降低部署门槛
- ✅ **事件驱动设计**天然适合做消息处理管道
- ✅ 可结合LLM做智能提取（如接入ChatGPT/通义千问等）

**与本项目的契合度：** 非常适合作为第一阶段的**消息监听+数据提取**核心引擎

## 三、推荐技术组合方案

### 方案一：纯Java技术栈（推荐 ★★★★★）

```
┌─────────────────────────────────────────────────────────┐
│                    技术架构（全Java）                      │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  ┌─────────────────────────────────────────────────┐    │
│  │              现有 RuoYi-Vue ERP                   │    │
│  │          (Spring Boot + MyBatis)                 │    │
│  │                                                  │    │
│  │   ┌──────────────┐  ┌──────────────────────┐    │    │
│  │   │  wecom-sdk   │  │ wework-webhook-      │    │    │
│  │   │  (企微API)   │  │ starter (群消息发送)  │    │    │
│  │   └──────┬───────┘  └──────────┬───────────┘    │    │
│  │          │                      │               │    │
│  │          ▼                      ▼               │    │
│  │   ┌────────────────────────────────────────┐   │    │
│  │   │      自建业务模块                        │   │    │
│  │   │  - 消息回调Controller                    │   │    │
│  │   │  - 数据提取规则引擎                      │   │    │
│  │   │  - 图表生成服务                          │   │    │
│  │   │  - 定时任务(Quartz)                      │   │    │
│  │   └────────────────────────────────────────┘   │    │
│  └─────────────────────────────────────────────────┘    │
│                                                          │
│  会话存档：通过企业微信官方HTTP API直接调用                │
│  （wecom-sdk中可能已包含会话存档相关接口，或单独对接）      │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

**优点：**
- ✅ 统一技术栈，维护成本低
- ✅ 直接复用ERP的用户体系、权限、日志等基础设施
- ✅ 团队学习成本最低
- ✅ 部署简单（只需一个Java应用）

**组件清单：**

| 组件 | 版本 | 用途 |
|------|------|------|
| wecom-sdk | latest | 企微API全量接入 |
| wework-webhook-starter | latest | 群消息自动发送 |
| 自建规则引擎 | - | 正则+NLP数据提取 |
| Apache POI | 已有 | Excel图表生成 |
| Quartz | 已有 | 定时任务调度 |

**注意：** 此方案需要确认wecom-sdk是否覆盖了**会话存档**相关接口。如未覆盖，需额外通过官方API对接或引入Go版WeWorkFinanceSDK作为辅助微服务。

### 方案二：混合技术栈（灵活 ★★★★☆）

```
┌──────────────────────────────────────────────────────────┐
│                  技术架构（Java + Python）                 │
├──────────────────────────────────────────────────────────┤
│                                                           │
│  ┌──────────────────┐    HTTP/API     ┌────────────────┐ │
│  │  Python 微服务    │◀──────────────▶│  Spring Boot   │ │
│  │                  │                 │  ERP 主系统     │ │
│  │ ┌──────────────┐ │                 │                │ │
│  │ │wecom_aibot_  │ │                 │ ┌────────────┐ │ │
│  │ │sdk (消息监听) │ │                 │ │ 数据接收    │ │ │
│  │ ├──────────────┤ │                 │ │ 清洗转换    │ │ │
│  │ │ 规则引擎/NLP  │ │                 │ │ 写入MySQL   │ │ │
│  │ ├──────────────┤ │                 │ │ 图表生成    │ │ │
│  │ │ 提取结果推送  │ │                 │ │ Webhook发送 │ │ │
│  │ └──────────────┘ │                 │ └────────────┘ │ │
│  └──────────────────┘                 └────────────────┘ │
│                                                           │
└──────────────────────────────────────────────────────────┘
```

**优点：**
- ✅ Python生态在NLP/AI方面更强
- ✅ wecom_aibot_sdk的WebSocket模式无需公网IP
- ✅ 消息处理逻辑独立演进，不影响ERP稳定性
- ✅ 可利用Python丰富的NLP库（jieba分词等）

**缺点：**
- ⚠️ 需要维护两套技术栈
- ⚠️ 多一个服务部署，运维复杂度增加
- ⚠️ 服务间通信需要考虑可靠性

**适用条件：** 团队有Python能力，且对消息提取准确率要求极高时选择。

## 五、快速启动建议

### 最小可行路径（MVP）— 2周出原型

```
Week 1: 环境搭建 + 消息通道打通
├── 注册企业微信 + 创建应用/机器人
├── 引入 wecom-sdk 或 wecom_aibot_sdk
├── 实现消息接收回调（能打印收到的消息）
└── 实现 Webhook 发送测试消息

Week 2: 核心闭环验证
├── 编写第一批数据提取规则（3-5个常见格式）
├── 实现提取→存储→生成Excel→发送的完整流程
├── 用真实消息样本测试准确率
└── 输出MVP演示版本
```

### 推荐起步顺序

```
1️⃣ 先做"发送"（最简单，立即见效）
   └── 用 wework-webhook-starter 实现定时推送测试消息
   
2️⃣ 再做"接收"（核心难点）
   └── 用 wecom-sdk 或 wecom_aibot_sdk 接入消息监听
   
3️⃣ 最后做"处理"（业务逻辑）
   └── 自建规则引擎 + 图表生成 + 数据存储
```
