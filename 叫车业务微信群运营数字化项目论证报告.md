## 一、项目概述

### 1.1 项目背景

富泉服装日常运营中存在大量叫车业务（原材料运输、成品发货、外发加工物流等），当前主要通过微信群进行运营管理，面临以下核心痛点：

| 痛点 | 现状描述 | 影响 |
|------|---------|------|
| **信息分散** | 叫车需求散布在多个微信群中，人工汇总耗时 | 每日需专人整理30-60分钟 |
| **响应不及时** | 司机无法及时看到新订单，响应延迟 | 平均等待时间15-30分钟 |
| **信息遗漏** | 消息被覆盖或忽略，订单漏接 | 月均遗漏率约5-8% |
| **统计困难** | 无法自动生成日报/周报，手工统计易出错 | 财务对账困难 |
| **追溯缺失** | 订单状态无记录，纠纷难处理 | 客户投诉增加 |

### 1.2 项目目标

| 目标维度 | 具体指标 |
|---------|---------|
| **自动化程度** | 微信群消息自动提取、解析、汇总 |
| **响应时效** | 新订单通知到司机时间 < 5分钟 |
| **准确率** | 信息提取准确率 ≥ 98% |
| **数据闭环** | 从微信群 → 自动化系统 → ERP全链路打通 |
| **效率提升** | 运营人员工作量减少 70%以上 |

### 1.3 项目范围

| 阶段 | 核心内容 | 交付物 |
|------|---------|--------|
| **第一阶段** | 微信群信息自动化处理系统 | 消息监听、数据提取、图表生成、自动发送 |
| **第二阶段** | ERP数据导入方案 | 数据接口、清洗转换、一致性校验 |
| **第三阶段** | 全链路业务迁移 | ERP下单模块、钉钉/微信集成、权限管理 |

### 2.2 技术选型论证

#### 2.2.1 微信消息获取方案对比

| 方案 | 技术路径 | 优点 | 缺点 | 可行性 | 推荐度 |
|------|---------|------|------|--------|--------|
| **A: 企业微信API** | 企业微信机器人Webhook / 会话存档API | 官方支持、稳定可靠、合规 | 需要企业微信环境、会话存档需付费认证 | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ 强烈推荐 |
| **B: 微信PC协议Hook** | 逆向Hook微信PC客户端协议 | 无需企业微信、可监听个人微信 | 违反微信协议、封号风险高、不稳定 | ⭐⭐ | ❌ 不推荐 |
| **C: 微信公众号/小程序** | 用户主动在公众号/小程序内提交 | 合规稳定、官方支持 | 改变用户习惯、需要用户配合操作 | ⭐⭐⭐⭐ | ⭐⭐⭐ 备选方案 |
| **D: 第三方平台对接** | 如企微管家、微伴助手等SaaS | 开箱即用、维护成本低 | 数据安全风险、费用较高、依赖第三方 | ⭐⭐⭐ | ⭐⭐⭐ 备选方案 |

**推荐方案：A - 企业微信API（主） + C - 微信小程序（辅）**

#### 2.2.2 企业微信技术路径详解

**方案A-1：企业微信机器人Webhook（轻量级）**

```
适用场景：单向通知推送（如将生成的清单推送到群）
技术实现：
  1. 创建企业微信群机器人，获取Webhook URL
  2. 后端调用Webhook API发送Markdown/图文消息
  3. 支持定时任务触发
限制：
  - 仅支持向群里发送消息，不能接收/读取群消息
  - 适合"自动发送清单"场景
```

**方案A-2：企业微信会话内容存档API（完整版）**

```
适用场景：双向消息读取 + 发送（完整的自动化处理）
技术实现：
  1. 申请企业微信「会话内容存档」权限（需付费认证）
  2. 配置可信服务器，接收回调数据
  3. 解密消息内容，提取结构化数据
  4. 通过API回复消息或发送到其他群
前提条件：
  - 已认证的企业微信（最低300元/年）
  - 开通会话内容存档功能（按成员收费）
  - 员工签署知情同意书
```

#### 2.2.3 消息监听机制设计

```python
# 消息处理流程伪代码
class WeChatMessageHandler:
    
    def on_message_received(self, message):
        # 1. 消息预处理
        raw_text = self.extract_text(message)
        
        # 2. 数据提取（基于规则引擎）
        extracted_data = self.rule_engine.extract(raw_text)
        
        # 3. 数据验证
        if not self.validator.validate(extracted_data):
            return self.handle_validation_error(extracted_data)
        
        # 4. 数据存储
        order_id = self.storage.save(extracted_data)
        
        # 5. 实时通知（可选）
        self.notifier.notify_drivers(extracted_data)
        
        return {"status": "success", "order_id": order_id}
```

### 3.3 数据清洗与转换规则

#### 3.3.1 数据映射关系

| 第一阶段数据 | ERP字段 | 转换规则 |
|-------------|---------|---------|
| `use_time` | `erp_car_order.use_time` | 标准化为YYYY-MM-DD HH:mm:ss |
| `from_addr` | `erp_car_order.from_address` | 地址标准化（匹配地址库） |
| `to_addr` | `erp_car_order.to_address` | 地址标准化（匹配地址库） |
| `cargo_type` | `erp_car_order.cargo_type` | 映射为标准枚举值 |
| `cargo_qty` | `erp_car_order.cargo_qty` | 数值标准化 |
| `contact` | `erp_car_order.contact_name` | 直接映射 |
| `phone` | `erp_car_order.contact_phone` | 格式校验（11位手机号） |
| `urgency` | `erp_car_order.urgency` | 映射枚举值 |
| `message_id` | `erp_car_order.source_msg_id` | 用于溯源 |
| `confidence` | `erp_car_order.extract_confidence` | 低于阈值标记待审核 |

#### 3.3.2 数据清洗规则

| 规则类型 | 规则描述 | 处理方式 |
|---------|---------|---------|
| **空值检查** | 必填字段不能为空 | 拒绝导入，返回错误提示 |
| **格式校验** | 手机号11位、时间格式正确 | 自动修正或拒绝 |
| **枚举映射** | 货物类型、紧急程度标准化 | 映射标准值，未知值标记 |
| **去重检查** | 同一消息ID不重复导入 | 幂等处理，跳过已存在记录 |
| **地址标准化** | 模糊匹配地址库 | 高置信度自动匹配，低置信度人工确认 |
| **时间合理性** | 用车时间不能是过去太久的时间 | 警告但不阻止 |

#### 3.3.3 数据一致性校验机制

```
校验层级：
Level 1 - 字段级校验（单个字段格式、长度、类型）
Level 2 - 记录级校验（字段间逻辑关系，如时间合理性）
Level 3 - 业务级校验（与已有数据的冲突检测）
Level 4 - 跨系统校验（与外部系统的数据一致性）

校验流程：
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 原始数据 │───▶│ L1字段校验│───▶│ L2记录校验│───▶│ L3业务校验│
└──────────┘    └──────────┘    └──────────┘    └────┬─────┘
                                                    │
                                            ┌───────▼───────┐
                                            │ L4跨系统校验   │
                                            │ (可选)         │
                                            └───────┬───────┘
                                                    │
                                            ┌───────▼───────┐
                                            │ 校验结果汇总    │
                                            │ 通过/待审/拒绝  │
                                            └───────────────┘
```

### 3.4 导入频率与策略

| 策略 | 频率 | 适用场景 | 实现方式 |
|------|------|---------|---------|
| **实时导入** | 每条消息处理后立即导入 | 需要即时可见的场景 | 消息处理完成后同步调用ERP接口 |
| **批量导入** | 每15分钟/每小时/每天定点 | 允许一定延迟的场景 | 定时任务批量聚合后导入 |
| **手动触发** | 用户点击按钮触发 | 需要人工确认的场景 | 前端操作触发导入 |

**推荐策略：实时导入为主 + 批量兜底为辅**

### 3.5 异常处理流程

| 异常类型 | 处理方式 | 通知对象 | 恢复机制 |
|---------|---------|---------|---------|
| **网络超时** | 本地重试3次，间隔递增 | 系统管理员 | 重试队列持久化 |
| **数据格式错误** | 记录错误详情，跳过该条 | 业务对接人 | 人工修正后重新导入 |
| **ERP接口异常** | 进入重试队列 | 开发人员 | 自动重试 + 告警 |

#### 5.6.2 关键风险深度应对

**风险1：消息提取准确率不达标（高风险）**

```
应对策略：
1. 分级处理机制：
   - 置信度 ≥ 95%：自动入库
   - 置信度 80%-95%：入库但标记"待审核"
   - 置信度 < 80%：不入库，转人工处理
   
2. 持续优化循环：
   收集误判案例 → 更新规则 → 重新训练 → 上线验证
   
3. 人工兜底：
   提供简易的人工修正界面，快速纠错
```

**风险2：司机用户抵触（高风险）**

```
应对策略：
1. 利益驱动：
   - 让司机看到好处（更快接到单、更清晰的信息、更快的结款）
   
2. 渐进式推广：
   - 先在1-2个积极司机试点 → 收集反馈 → 优化体验 → 全面推广
   
3. 降低门槛：
   - 小程序操作尽量简单（3步以内完成核心操作）
   - 保留微信群作为备用通道
   
4. 激励措施：
   - 接单响应快的司机优先派单
   - 评分高的司机获得更多机会
```

**风险3：企业微信政策变化（中高风险）**

```
应对策略：
1. 技术解耦：
   消息采集层与业务逻辑层分离，更换采集渠道不影响核心系统
   
2. 备选方案就绪：
   - 方案B：微信公众号/小程序提交
   - 方案C：钉钉群机器人
   - 方案D：独立的H5表单页面
   
3. 数据主权：
   核心数据存储在自有ERP系统中，不依赖任何第三方平台
```
