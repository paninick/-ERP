## 一、如何有效使用我的能力

### 1.1 核心沟通模式

在未来的协作中，使用以下模式可以获得最佳效果：

```
"请 [动作] [目标]，满足 [具体要求]"
```

**示例**：
- ✅ "请重构认证模块，使用 JWT 替代 Session Cookie"
- ✅ "创建一个用户管理的 REST API 端点，包含 CRUD 操作"
- ✅ "分析数据库查询的性能瓶颈并提出优化建议"

### 1.2 获得最佳结果的最佳实践

| 实践 | 说明 | 好处 |
|------|------|------|
| **提供上下文** | 分享相关文件、错误信息或代码片段 | 我能理解完整情况 |
| **具体明确** | 清晰说明需求、约束和目标 | 无歧义，更快得到结果 |
| **分解任务** | 将大任务拆分成可管理的小块 | 更好的专注度，更高质量 |
| **要求解释** | 请求建议背后的推理 | 学习并理解解决方案 |
| **迭代反馈** | 审查每个迭代并提供反馈 | 持续改进 |

### 1.3 有效的命令模式

#### 代码分析
```
"分析这个文件，找出：
1. 潜在的 bug 或问题
2. 性能瓶颈
3. 安全漏洞
4. 代码质量改进建议"
```

#### 重构
```
"重构这段代码，以：
1. 提高可读性和可维护性
2. 遵循 [特定框架/风格指南]
3. 添加适当的错误处理
4. 编写单元测试"
```

#### 实现
```
"实现 [功能]，满足这些要求：
1. [需求 1]
2. [需求 2]
3. [约束 1]
4. 遵循 [架构/模式]"
```

#### 调试
```
"我遇到了这个错误：[错误信息]
这是相关代码：[代码片段]
预期行为是：[预期]
请帮助诊断并修复。"
```

### 1.4 完整的问题描述模板

```
"这是我想做的：[目标]
这是我现有的：[当前代码/文件]
这是我尝试过的：[之前的尝试]
这是遇到的问题：[具体问题]
请帮我：[具体请求]"
```

## 二、集中式智能代理系统使用指南

### 2.1 系统概述

我们已经创建了一个完整的集中式智能代理系统，基于 `learn-coding-agent` 的 12 层架构实现。

**文件位置**：`d:\erp\centralized_agent_system.py`

**核心功能**：
- ✅ 会话管理（创建、暂停、恢复、完成）
- ✅ 消息持久化（JSONL 格式，崩溃可恢复）
- ✅ 任务管理（创建、更新、查询）
- ✅ 上下文压缩（s06 层）
- ✅ 统计信息
- ✅ 会话导出

### 2.2 快速开始

#### 基本使用

```python
from centralized_agent_system import CentralizedAgentSystem, TaskStatus

# 1. 初始化系统
system = CentralizedAgentSystem("./agent_sessions")

# 2. 创建新会话
session_id = system.create_session(
    project_name="我的项目",
    initial_context={
        "language": "Python",
        "framework": "FastAPI",
        "goal": "实现某个功能"
    }
)

# 3. 添加消息
system.add_user_message(session_id, "请帮我实现这个功能")
system.add_assistant_message(session_id, "好的，我来帮你...")

# 4. 创建任务
task_id = system.create_task(session_id, "设计数据库 schema", priority=1)

# 5. 更新任务状态
system.update_task_status(session_id, task_id, TaskStatus.COMPLETED)

# 6. 获取统计信息
stats = system.get_session_stats(session_id)

# 7. 列出会话
sessions = system.list_sessions(10)
```

### 2.3 API 参考

#### 会话管理

| 方法 | 说明 |
|------|------|
| `create_session(initial_context, project_name)` | 创建新会话 |
| `get_session(session_id)` | 获取会话 |
| `list_sessions(limit, state)` | 列出会话 |
| `pause_session(session_id)` | 暂停会话 |
| `resume_session(session_id)` | 恢复会话 |
| `complete_session(session_id)` | 完成会话 |

#### 消息管理

| 方法 | 说明 |
|------|------|
| `add_message(session_id, role, content, metadata)` | 添加消息 |
| `add_user_message(session_id, content, metadata)` | 添加用户消息 |
| `add_assistant_message(session_id, content, metadata)` | 添加助手消息 |

#### 上下文管理

| 方法 | 说明 |
|------|------|
| `update_context(session_id, key, value)` | 更新上下文 |
| `set_context(session_id, context)` | 设置完整上下文 |
| `compress_context(session_id, max_messages, keep_recent)` | 压缩上下文 |

#### 任务管理

| 方法 | 说明 |
|------|------|
| `create_task(session_id, description, priority)` | 创建任务 |
| `update_task_status(session_id, task_id, status, notes)` | 更新任务状态 |
| `get_tasks(session_id, status)` | 获取任务列表 |

#### 其他功能

| 方法 | 说明 |
|------|------|
| `get_session_stats(session_id)` | 获取会话统计 |
| `export_session(session_id, output_file)` | 导出会话 |

### 2.4 数据存储格式

系统使用两种文件格式存储数据：

#### 1. JSONL（消息日志）
```
session_xxx.jsonl
```
- 追加式日志格式
- 用户消息使用阻塞写入（fsync）保证崩溃可恢复
- 每一行是一个 JSON 对象

#### 2. JSON（元数据）
```
session_xxx_meta.json
```
- 包含会话状态、上下文、任务列表等
- 完整的 JSON 格式，便于读取
