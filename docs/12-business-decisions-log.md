> **定位**:凡是**技术团队无法自行拍板、必须由业务 Owner / 合规 / 法务 / 客户回复**的决策点,都在本档建立一条记录。跟踪"提问 → 等待 → 回复 → 实施"的完整生命周期。
> **产出日**:2026-04-22 · **版本**:v1.0
> **维护人**:PM(主)+ 技术负责人(协)
> **更新规则**:每条决策有实质变化(提出 / 答复 / 实施 / 撤销)时更新对应条目 + 索引表 + 变更日志

## 0. 使用说明

### 0.1 为什么需要这份台账

- **docs/00 §5.2 P0 清单** 是 **索引**(一行一问)
- **本档 docs/12** 是 **工作台账**(每条决策完整上下文 + 选项 + 回复证据 + 影响)
- 二者配合:新人 15 分钟看完 00 掌握全局,深入某条时来 12 查详情

### 0.2 决策 ID 规则

```
BD-{3位序号}  (Business Decision)
从 BD-001 开始递增,不复用,已取消的决策保留编号
```

### 0.3 状态流转

```
🟡 待回复  →  ✅ 已决策  →  🔧 实施中  →  🟢 已闭环
              ↓
           ❌ 已取消 / 冻结
```

### 0.4 回复证据要求

业务 Owner / 合规 / 法务的回复**必须有证据留存**,不得仅凭口头:
- **邮件正文 + 日期 + 签名**(推荐,可直接引用)
- **微信 / 钉钉截图 + 发送时间**(次选,注意 PII 脱敏)
- **会议纪要 + 参会人 + 日期**(第三选)
- **合同 / 法律文件正本扫描件**(法律级决策必用)

### BD-001 · 业务单号是否允许跳号

| 字段 | 值 |
| :-- | :-- |
| **状态** | 🟡 待回复 |
| **类型** | 业务规则决策 |

| **提出人** | 技术团队(Claude + 架构师) |
| **答复人** | 业务 Owner |
| **回复截止** | 2026-04-25(本周内)|
| **影响范围** | P3.7 `BillNoGenerator` 集成与返工风险 · `docs/11 Assumption 5` |
| **触发来源** | commit `685c9b0c refactor(erp): BillNoGenerator 改用 StringRedisTemplate` + Codex PR review brief v3 |

#### 2.1.1 一句话问题

**销售单 SO / 生产工单 JOB / 采购单 PO** 的单号,是否允许出现"空洞"(如 SO-001、SO-002、**缺 SO-003**、SO-004)?

#### 2.1.2 上下文

- 系统并发开单、异常回滚、失败重试等场景下,单号可能被"占用但未最终落库",形成空洞
- 这在技术上是**常见且可接受**,但要不要接受是**业务决策**,非技术决策
- 不同答案对应不同实现路径,工期与性能差异大

#### 2.1.3 三选项对比

| 选项 | 业务规则 | 实现方式 | 并发性能 | 开发成本 | 适合 |
| :-: | :-- | :-- | :-: | :-: | :-- |
| **A** 全部允许跳号 | SO-001、002、**004**(跳 003 无事)| Redis 序号 | 快 | 低 | 大多数 ERP 默认 |
| **B** 全部不允许跳号 | 必须 001、002、003、004 连续 | 数据库锁表 | 慢 | 高 | 极少数合规场景 |
| **C** 按类型分 | SO 允许,JOB 不允许(例)| 混合 | 中 | 中 | 特定业务要求 |

#### 2.1.4 行业常识参考

- **中国增值税发票号** `invoice_no`:法律硬要求连续 ← **单独处理,不在本条范围**
- **销售 / 采购 / 生产单据号**:金蝶、用友、SAP 默认**允许跳号**
- **日方客户 CoC / RSL**:暂未发现对单据号连续性的要求(若有请告知,见 BD-004)
- **审计问"某个号去哪了?"**:标准答复"系统分配后业务失败,已作废号,有日志可查"

#### 2.1.5 工期影响

| 选项 | BillNoGenerator 返工 | P3 总工期 |
| :-- | :-: | :-: |
| A | 0 人日(当前实现已满足) | 72 人日 |
| B | +3 人日 | 75 人日 |
| C | +1-2 人日 | 73-74 人日 |

#### 2.1.6 技术补充

- 当前 `BillNoGenerator`(commit `685c9b0c`)用 Redis `INCR` 原子自增,性能好但天然允许跳号
- 若选 B/C,需新增 `sys_bill_sequence` 数据库序列表 + 分布式锁
- Codex review brief v3 的 Assumption 5 默认"允许跳号"作为 Redis INCR 不作为 P0 的前置条件

#### 2.1.7 推荐回复格式(业务 Owner 使用)

```
□ A:全部允许跳号(推荐)

□ B:全部不允许跳号
   理由:________________

□ C:以下单号不允许跳号,其他允许:
   □ 销售单 SO     理由:________________
   □ 生产工单 JOB  理由:________________
   □ 采购单 PO     理由:________________
   □ 其他:__________

签名 / 日期:________________
```

| 选项 | 动作 |
| :-- | :-- |
| A | 关闭 BD-001,Codex review brief Assumption 5 确认,PR review 启动 |

#### 决策组件映射

| 原方案 | 新方案 | 影响工作包 | 工期变化 |
| :-- | :-- | :-- | :-: |
| MinIO(自建对象存储) | **阿里云 OSS** | P3.7.3 | 2d → 1d(-1)|
| 本地 AES + 自研 KMS | **阿里云 KMS SDK** | P3.5.1 | 3d → 1d(-2)|
| 自建 MySQL 5.7 | **RDS for MySQL 5.7**(默认 utf8mb4)| P3.8.1 / P6.1 | 减负(DBA 0.5→0.2)|
| 自建 Redis 5+ | **Redis Tair**(阿里云托管) | P3.7 WebSocket 会话 + JWT | 运维简化 |
| 自建 Nginx | **SLB 负载均衡**(阿里云) | P6.0 Docker | 同时绕开 nginx 老版本问题 |

#### 组件选型细节

- **OSS**:标准存储 + CDN 回源;样品图/色卡用私有桶 + 预签名 URL
- **KMS**:RSA 非对称密钥做应用层数据加密(身份证/银行卡);对称密钥做 AES-GCM
- **RDS**:推荐高可用版(主备),性能规格按 Pilot 基线 2c4g 起步
- **Tair**:推荐标准版 Redis 社区 6.0 系列;如需 Stream 用 Tair 增强版
- **SLB**:应用型 ALB(非 NLB),内网 + 公网双实例(按需)
