# 99 · 假设登记册（Assumptions Ledger）

> **定位**：项目在缺乏部门访谈纪要、U8 schema、JIS / RSL 文件、日方客户 CoC 等原始材料的前提下产出了 5 份战略交付物。本登记册把散落在 01-05 各章的**所有明确假设**统一归档，明示校正路径与更新策略，防止假设静默变成事实。
> **产出日**：2026-04-21 · **版本**：v1.0
> **维护节律**：每周五 PM 更新，获得新材料后立即校正

---

## 1. 如何使用本登记册

### 1.1 假设分类

| 类型 | 定义 | 行动 |
| :-- | :-- | :-- |
| **A 行业通识假设** | 基于针织外贸行业通用实践 | 与业务 Owner 确认 |
| **B 代码现状假设** | 基于探盘报告的推测 | 与架构师校验 |
| **C 法规条款假设** | 基于公开法律条文 | 法务复核 |
| **D U8 字段假设** | 基于 U8 12/13.x 标准模型 | 拿到账套 schema 后校正 |
| **E 规模假设** | 基于前期论证报告的 2-3 工厂 | 运营数据校正 |
| **F 时间假设** | 基于项目估算 | 实际进度校正 |

### 1.2 状态定义

| 状态 | 含义 |
| :-- | :-- |
| 🟡 **待校正** | 当前是工作假设，尚未拿到实际数据 |
| 🟢 **已校正** | 获得真实数据，假设被替换为事实 |
| 🔴 **已推翻** | 真实数据与假设冲突，需修订相关交付物 |
| ⚪ **可保留** | 实际情况支持此假设继续使用 |

### 1.3 更新流程

1. **获取材料** → PM 登记新材料
2. **比对假设** → 架构师逐条对比
3. **判定状态**：
   - 符合 → ⚪ 可保留
   - 差异小 → 更新假设条目为"已校正"，注释差异
   - 差异大 → 🔴 推翻，发起相关交付物修订
4. **更新下游文档**（如果需要）

## 6. 推翻后的处理流程（示例）

**示例**：拿到 U8 schema 后发现 AS-033（EAI 接口可用）🔴 推翻。

1. **即时更新**：
   - `99 §2` 中 AS-033 状态改 🔴
   - 变更日志加一条
   - 相关交付物 `04 §4.3`、`05 P4` 改注释

2. **启动备案方案**：
   - 评估场景 B（U8 全量退役）
   - 或：集成商替代方案（找第三方）
   - CCB 评审

3. **WBS 调整**：
   - P4 阶段重规划，工期可能 +2-3 个月
   - 预算重估
   - 风险册 T04 升级为"已发生问题"

4. **沟通**：
   - 向决策层专题汇报
   - 更新 WBS 日程
   - 调整 MS2 里程碑日期
